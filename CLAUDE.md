# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A Unity 6 (`6000.3.0f1`, URP) project named **HorizontalChild** whose sole purpose is to host the **Editor Terminal** asset by Bitpurr Digital (currently v1.4.2). Everything outside `Assets/Bitpurr Digital/` (the URP settings, sample scene, tutorial Readme asset) is the default Unity 6 template and not the subject of work. Treat `Assets/Bitpurr Digital/Editor Terminal/` as the product.

## Building, running, "testing"

There is no CLI build, no test framework wired up, and no CI. Workflow is entirely inside the Unity Editor:

- Open the project with Unity Hub using exactly version `6000.3.0f1` (see `ProjectSettings/ProjectVersion.txt`).
- Launch the terminal via the menu `Window > Editor Terminal` (registered by `TerminalEditorWindow.ShowWindow`).
- The `.csproj` / `.sln` at the repo root are regenerated by Unity from `EditorTerminal.asmdef` — do **not** hand-edit them; modify the `.asmdef` instead.
- The Editor Terminal's own help is the closest thing to a manual: type `et help` (or `editorterminal help`) inside the running terminal window.

When asked to "run tests" or "build", there is nothing to run — say so rather than inventing a command.

## Architecture

### Assembly boundary

All product code lives under `Assets/Bitpurr Digital/Editor Terminal/Editor/` and compiles into the `EditorTerminal` assembly defined by `EditorTerminal.asmdef`. The asmdef restricts the assembly to `includePlatforms: ["Editor"]` — this code is never shipped in player builds. Adding new files outside this folder will put them in the default `Assembly-CSharp(-Editor)` assemblies, which is almost certainly wrong for this codebase.

### The Unity 5/6 split (most important thing to understand)

The same source file targets both Unity 2022.3+ (pre-UI Toolkit data-source API, called "Unity 5" in the code) and Unity 6+ (new `dataSource` / `DataBinding` API). The split is mediated by `#if UNITY_6000_0_OR_NEWER` and shows up in three coupled places — change one, change all three:

1. **`TerminalEditorWindow.cs`** — top of file aliases `HistoryBufferUIElement` to `TextElement` (U6) or `TextField` (pre-U6). The history buffer is read-only display in U6 and an editable-but-bound `TextField` in older Unity.
2. **`TextArea.cs`** — partial `UxmlElement` (U6) vs `UxmlFactory`/`UxmlTraits` pattern (pre-U6) for registering the custom element used by the UXML.
3. **UXML files** — `TerminalUI_Unity6.uxml` uses `<ui:DataBinding ... data-source-path="historyBuffer">`, `TerminalUI_Unity5.uxml` uses legacy `SerializedObject.Bind` plumbing in `SetupDataBindings()`. `GetUxmlAssetPath()` picks one at compile time.

There is similar `#if UNITY_EDITOR_WIN` vs Unix branching for the shell selection (`powershell.exe`/`cmd.exe` vs `/bin/bash` etc.) and git executable auto-discovery in `LocateGitExecutable()`.

### Command execution flow

`TerminalEditorWindow.RunCommand` is the choke point. It dispatches in this order:

1. **Built-ins handled in-process**: `clear`/`cls` (just clears the history buffer), `cd [path]` (mutates `_path`, updates the chevron, never spawns a process). Note that `cd` is intercepted by string match — adding new in-process built-ins means adding another branch here, not editing the external shell.
2. **Internal `editorterminal` / `et` commands**: routed to `RunInternalCommand`, which is a `switch` on the first arg covering `set_git_location`/`sgl`, `set_shell`/`ss`, `set_font_size`/`sfs`, `set_font_color`/`sfc`, `set_background_color`/`sbc`, `set_default_colors`/`sdc`, `config`/`c`. Adding a new internal command means: add cases for both the long and short name in that switch, and add a line to `RenderHelp()`.
3. **Git fast path**: if the command starts with `git ` and `GitLocation` (an `EditorPrefs` key) is set, the binary is invoked directly without going through a shell — this exists for speed and to avoid PATH issues.
4. **Everything else** goes through the external shell selected by the `ExternalShell` `EditorPrefs` key.

### Async output → main-thread UI

`System.Diagnostics.Process` raises `OutputDataReceived` / `ErrorDataReceived` on background threads. UI Toolkit is main-thread-only, so output is funneled through `_outputBuilder` (a `StringBuilder` guarded by `lock`) and `_uiUpdateQueue` (a `Queue<Action>` also guarded by `lock`). `OnUpdate`, hooked to `EditorApplication.update`, drains the queue on the main thread, appends to `_terminalData.historyBuffer`, calls `SetHistoryBufferUI`, and scrolls to the bottom. If you add a feature that touches UI from a process callback, route it through `Enqueue(...)` — do not touch `rootVisualElement` directly from those threads.

### Persistence

Two layers, both using `EditorPrefs`:

- **User configuration** (font, colors, git location, shell) — each setting is a property with a get/set on `EditorPrefs.GetX($"{TerminalName}_<Key>")`. `TerminalName` is the constant `"Editor Terminal"` and is part of every key, so renaming it silently invalidates user settings.
- **Window state** (`historyBuffer`, `commandBuffer`) — serialized as JSON via `JsonUtility` into the single key `"{TerminalName}_WindowData"` in `SaveTerminalData()` (called from `OnDisable`) and rehydrated in `LoadTerminalData()` (called from `CreateGUI`).

There is no settings UI — all configuration is done by typing internal commands into the terminal itself.

### Asset path resolution

`GetBasePath()` does **not** hardcode the install path. It calls `AssetDatabase.FindAssets("Bitpurr Digital")`, filters to folders literally named that, then searches them for the `TerminalEditorWindow` script and returns its directory. This is what makes the asset relocatable inside a user's project, so keep using `GetBasePath()` (or the `Unity6UxmlPath` / `Unity5UxmlPath` / `IconPath` constants derived from it) when referencing files in this folder — never hardcode `Assets/Bitpurr Digital/...`.

### Keyboard handling

`KeyDownEvent` is registered with `TrickleDown.TrickleDown` because the embedded `TextField` would otherwise swallow Ctrl/Cmd combinations. Shortcuts implemented: `Ctrl/Cmd + L` (clear), `U` (clear command line), `K` (kill-to-end), `E` (end of line, U2022.3+), `R` (cycle previous commands), `Z` (kill running process), `+`/`-` (font size). Tab completion is implemented through the `RegisterValueChangedCallback` on the command field (intercepts the literal `\t` Unity inserts) rather than via `KeyDownEvent`.
