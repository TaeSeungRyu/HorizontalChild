# 어린이 대항해 시대 — 준비 문서

> 어린이가 **대항해시대 II**를 모티브로 한 세계 항해를 통해 **세계 지리 / 지역 특산물 / 숨겨진 발견물 / 역사** 를 자연스럽게 배우는 교육용 게임.

이 문서는 본격 구현 전 정해두어야 할 사항과 자산 / 코드 구조를 모아둔다. 구체적인 게임 디자인(레벨, 시나리오, 발견물 목록 등)은 별도 디자인 문서에서 다룬다.

## 1. 비전 (확정)

| 항목 | 결정 |
|---|---|
| 장르 | 교육용 항해 어드벤처 |
| 모티브 | 코에이 『대항해시대 II』 |
| 타겟 연령 | **초등 저학년 (1~3학년)** |
| 시각 | **2D 베이스 + 약간의 3D (2.5D)** |
| 조작 | **직접 조종** (자동 길찾기 아님) |
| 시대 배경 | **실제 대항해시대 (15~17세기)** — 가상 세계 아님 |
| 플랫폼 | **Android 전용** (iOS·PC 빌드 안 함) |
| 기준 기기 | **Galaxy S23** (2340×1080, 19.5:9, **가로 모드 고정**) |
| 1차 시연 항구 | **리스본 ↔ 세우타** |
| 아트 톤 | **화려하지 않은 미니멀 / 로우폴리 / 플랫** — CC0·OFL 우선 |
| 앱 정책 | **오프라인 · 권한 0 · 광고 0 · IAP 0 · 비상업** (순수 교육 목적) |
| 배포 채널 | **Google Play 정식 출시** (대상 연령: 만 9~12세 + 만 13세 이상 혼합 또는 만 13세 이상 — §11.5) |
| 플레이 길이 | **긴 모험** (점진적 진행 · 자동 저장 · 도감 누적) |
| 선택 국적 | **8개국** — 포르투갈 / 네덜란드 / 영국 / 스페인 / 이탈리아 / 오스만 / 조선 / 중국 |
| 교육 목적 | ① 세계 지리 ② 지역 특산물 ③ 숨겨진 발견물 ④ 역사 |

핵심 원칙: **시뮬레이션 깊이보다 접근성 우선.** 어린이가 5~10분 안에 첫 항해를 시작할 수 있어야 한다.

### 1.1 결정이 시스템에 끼치는 영향 (요약)
- **초등 저학년**:
  - 모든 텍스트는 받침 많은 한자어 / 어려운 외래어 회피. 발음하기 어려운 지명(예: "캘리컷")은 한글 음차 + 작은 해설.
  - 폰트는 크고 둥글게, 터치/클릭 타깃은 넉넉히.
  - 실패(좌초·길 잃음)에 페널티를 주지 않는다. "잠시 후 자동 복귀" 같은 부드러운 처리.
  - 한 발견물의 해설은 **2~4문장**을 넘기지 않는다.
- **직접 조종**:
  - 클릭 자동 항해 시스템은 **만들지 않는다**. 모든 이동은 플레이어 입력 기반.
  - 조작은 단순화: 좌/우 = 키 회전, 위 = 가속, 아래 = 감속. 마우스 보조는 옵션.
  - 약한 관성(스르륵 멈춤)으로 "배 같은 느낌"은 주되, 멀미 날 정도는 X.
  - 미니맵 + 항구/발견물 방향 표시기로 길을 잃지 않게 보조.
- **실제 대항해시대 (15~17세기)**:
  - 모든 항구·발견물·인물·사건은 사실 기반. 출처를 데이터에 기록.
  - 시대상 민감 주제(노예무역, 식민지화, 종교 갈등)는 회피하지 않되 **사실 중심·중립적 어조**로.
  - 시각 톤은 고지도·범선 일러스트 분위기. 너무 SF / 판타지로 가지 않는다.
- **Android 전용 (Galaxy S23 기준)**:
  - 입력은 **터치 only** — 가상 조이스틱, 슬라이더, 큰 탭 버튼. 키보드/마우스 매핑 만들지 않음.
  - 렌더 파이프라인은 **`Mobile_RPAsset` / `Mobile_Renderer`** 사용 (`PC_*` 에셋은 빌드에서 제외).
  - **가로 모드 고정** (Landscape Left/Right). Player Settings에서 세로 비활성화.
  - **기준 해상도 2340×1080 (19.5:9)**. Canvas Scaler `Scale With Screen Size`, Reference Resolution **2340×1080**, Match `Width or Height = 0.5`.
  - 안전 영역: 가로 모드에서 펀치홀은 **상단 가장자리(시계 기준 12시 → 가로에선 좌 또는 우 끝)** 에 위치. 중요 UI는 펀치홀을 침범하지 않도록 `Screen.safeArea` 기반 패딩.
  - 성능 예산: **S23 (Snapdragon 8 Gen 2, 8GB RAM)** 기준이므로 매우 여유로움. 60fps 안정 + 적당한 셰이더/파티클 여유. 그래도 발열·배터리 배려해 항구 화면 등 정적인 곳에서는 프레임 캡 또는 일시정지 가능 구조.
  - **개발 환경**: Unity Hub의 Add Modules로 **Android Build Support + OpenJDK + Android SDK & NDK Tools** 설치. Mac/Xcode 불필요.
  - **빌드 산출물**: Google Play 배포는 `.aab` (Android App Bundle), 사이드로드 테스트는 `.apk`. 1차 시연은 USB로 S23에 `.apk` 설치하는 흐름.
  - **Player Settings**: Scripting Backend = **IL2CPP**, Target Architectures = **ARMv7 + ARM64** (Play Store는 ARM64 필수), Minimum API Level = 23 이상 권장, Target API Level = 최신.
  - **키스토어**: 1차 시연 단계엔 Unity 기본 debug.keystore로 OK. **Play 정식 출시 결정됨** → 빠른 시점에 release용 키스토어를 생성하고 안전한 곳에 백업·이중화. 분실 시 동일 패키지명으로 업데이트 영구 불가. Play App Signing 사용 권장.
- **긴 모험 (Play 정식 출시 전제)**:
  - 한 세션에서 끝나는 게임이 아니므로 **자동 저장은 필수**. 항구 입항 / 발견물 획득 / 큰 메뉴 진입 시점마다 저장.
  - 진행 시스템: 점수·경험치·체력 대신 **"도감을 채워간다 + 새 지역의 잠금이 풀린다"** 가 진행감의 핵심.
  - 콘텐츠 분량 기준: 최소 30~50개 항구 + 100개 안팎 발견물을 **데이터 구조가 받아낼 수 있어야** 함 (1차 마일스톤에 실제로 다 만들지는 않더라도).
  - "이어하기" / "처음부터" — 슬롯은 어린이용이라 1~3개 정도로 단순하게.
- **8개국 선택 (포르투갈 · 네덜란드 · 영국 · 스페인 · 이탈리아 · 오스만 · 조선 · 중국)**:
  - 첫 화면에서 국적을 고르면 **출발 항구 · 시작 인사말 · 초기 도감 상태**가 달라진다.
  - 데이터 모델에 **`NationData` ScriptableObject** 를 처음부터 두어, 항구·발견물 데이터가 국적을 참조할 수 있게 한다.
  - **1차 마일스톤은 포르투갈/리스본 고정**. 국적 선택 UI는 2차 마일스톤. 그래도 데이터 구조는 처음부터 8개국을 가정해 설계.
  - 교육 균형: 유럽 5개국이 동시대 비유럽 3개국(오스만·조선·중국)과 어떻게 교류했는지 자연스럽게 드러나도록 콘텐츠를 구성.

## 2. 대항해시대 II에서 가져올 요소 (그리고 가져오지 않을 것)

원작은 좋은 모티브이지만 그대로는 너무 무겁다. 아래와 같이 **선택적으로 차용**한다.

### 가져올 요소
- **세계 지도 항해**: 전 세계 단일 지도 위에서 항구 간을 오가는 큰 그림.
- **항구**: 도착 시 메뉴가 열리고, 도시 일러스트 + 그 지역 정보 표시.
- **특산물 / 무역 (간소화)**: 각 항구의 대표 특산물을 보여주는 학습 카드. 1차에서는 "구매-판매로 돈 벌기" 보다 **"이 지역의 특산물을 만나봐!"** 에 가깝게.
- **발견물 시스템 (핵심 교육 장치)**: 특정 지점에 도달하거나 항구에서 의뢰를 받아 **랜드마크 / 동식물 / 역사적 사건 / 유적** 을 발견 → 발견 시 일러스트 + 해설 텍스트(어린이 눈높이) + "발견물 도감"에 등록.
- **항해일지 / 도감**: 발견한 항목과 방문한 항구를 모으는 컬렉션 UI. 진행도 가시화.

### 가져오지 않을 (또는 1차에서 빼는) 요소
- 복잡한 무역 시세 변동, 금리, 투자 시스템.
- 해전 / 함대 전투 (어린이 친화성과 충돌).
- 항해사 고용, 함대 관리.
- 항해 중 식량/물/괴혈병 관리 (스트레스 요소 제거).

→ 1차 구현은 **"세계지도 항해 + 항구 도착 + 발견물"** 트라이앵글에 집중.

## 3. 2.5D — 무엇을 어떻게 섞나

"2D 베이스 + 약간 3D" 를 구체화하기 위한 결정.

### 권장 방향
- **세계지도 항해 화면**: 카메라는 top-down에 가까운 각도(예: 60~75°)로 살짝 기울임. 지도는 큰 3D 평면 + 양각으로 약간 돌출된 산맥/대륙. 배는 **3D 로폴리 모델**로 키 회전과 약간의 기울기(파도 흔들림) 표현.
- **항구 화면**: 2D 일러스트 배경 + 2D UI 카드. 굳이 3D 안 써도 됨.
- **발견 이벤트**: 2D 일러스트 컷씬 + 카메라 살짝 줌/팬으로 입체감.

### 기술 선택
- **렌더 파이프라인**: 기존 URP 유지. URP는 2D/3D 혼합에 적합.
- **카메라**: Cinemachine 사용 권장 — 항해/항구/이벤트별 가상 카메라를 정의하고 부드럽게 전환.
- **스프라이트 vs 3D 모델**:
  - 배: 저폴리 3D (회전이 부드럽고 파도 흔들림 표현 쉬움).
  - 지도 위 항구 마커, UI: 2D 스프라이트.
  - 대륙: 2.5D 평면에 노멀맵 또는 약간의 메쉬 양각.
- **2D 라이트**: URP의 2D Renderer 또는 3D Renderer 위에 Sprite Lit 셰이더 — 처음에는 단순하게 가고, 분위기 잡을 때 도입.

→ **`Mobile_RPAsset` / `Mobile_Renderer` 기준** 으로 시작한다. 기존 `PC_*` 에셋은 참고용으로 두되 빌드에는 사용하지 않음. 빌드 타깃은 **Android 단독** (Google Play 정책에 맞춰 ARM64 빌드 포함).

## 4. 게임 루프 (1차 마일스톤 범위)

```
[세계지도] → 항구 선택/항해 → [항해 중] → 항구 도착 또는 발견물 트리거
    ↑                                              ↓
    └──────  [항구 화면] / [발견 이벤트] ←────────┘
                       ↓
                  [도감에 등록]
```

### 1차 마일스톤: "리스본에서 출발해 세우타까지 항해하고, 도중에 지브롤터 해협을 발견한다"

**왜 리스본 ↔ 세우타인가?**
- 1415년 포르투갈의 **세우타 정복**은 대항해시대의 실질적 시작점 — 게임의 첫 항해로서 역사적 의미가 깊다.
- 약 600km 의 짧은 항로 → 초등 저학년이 처음 항해하기에 부담 없는 거리.
- 사이에 **지브롤터 해협**이 있어 자연스러운 첫 발견물 위치.
- 지중해와 대서양의 경계 → 이후 지중해 / 대서양 두 방향으로 확장 가능.

**체크리스트**
- [ ] `Assets/Game/` 폴더 + `Game.asmdef` 생성
- [ ] 빌드 타깃을 Android (또는 iOS) 로 전환, `Mobile_RPAsset` 적용
- [ ] `World.unity` 씬: 이베리아 반도 + 북아프리카 일부가 보이는 2.5D 평면 + Cinemachine 카메라 (가로 모드)
- [ ] 배 3D 모델 (임시 캐러벨선 또는 저폴리 임시) + `ShipController` (터치 입력으로 조타/추진)
- [ ] 터치 UI: 가상 조이스틱(조타) + 가속/감속 슬라이더 또는 버튼
- [ ] 항구 트리거 2개: **리스본**(38.7°N, 9.1°W 근방), **세우타**(35.9°N, 5.3°W 근방)
- [ ] 발견물 트리거: **지브롤터 해협** (36.0°N, 5.6°W) + 발견 패널 UI (일러스트 + 2~4문장 해설)
- [ ] 도감 ScriptableObject + 발견 시 등록 + 도감 보기 UI (탭으로 열림)
- [ ] 항구 도착 화면: 임시 배경 + "리스본 / 포르투갈" "세우타 / 모로코" 라벨 + 특산물 한 줄
- [ ] 실기 1대(안드로이드 또는 아이패드)에 빌드 → 손가락으로 실제 조작감 확인

이 한 사이클이 굴러가면 그 위에 항구/발견물/지역을 **데이터(SO) 추가만으로** 늘려갈 수 있다.

## 5. 교육 콘텐츠 — 데이터로 다룰 것

게임 시스템과 별개로, 교육용 게임은 **콘텐츠 데이터의 양과 질**이 핵심이다. 처음부터 데이터 구조를 깔끔히 잡아두면 나중에 콘텐츠 추가가 코드 변경 없이 가능하다.

### 5.1 데이터 모델 (ScriptableObject로)

> ⚠ 2026-05-29 업데이트: 사용자가 게임 메카닉 기획(능력치·전투·NPC·도시 기능·미션·시세)을 추가했다. 자세한 시스템 규칙은 **[`Assets/Game/Data/GAME_MECHANICS.md`](Assets/Game/Data/GAME_MECHANICS.md)** 참조. 아래는 그에 맞춰 확장된 데이터 모델 요약.

콘텐츠 SO (1차 마일스톤 우선):
- **`NationData`** (플레이어가 고르는 국적): 국가 이름(한글/원어), 깃발 아이콘, **시작 항구(PortData 참조)**, 시작 인사말 / 짧은 소개, UI 강조 색상(선택). 처음부터 8개국 만들어둘 것: 포르투갈 · 네덜란드 · 영국 · 스페인 · 이탈리아 · 오스만 · 조선 · 중국.
- **`PortData`**: 이름, 위/경도, 도시 일러스트, **소속 국가(NationData 참조)**, 한 줄 설명.
- **`PortFacilities`** (항구당 1개): 시장·모험가조합·조선소·광장·항구 5개 기능 보유 여부 + 그 안의 카탈로그.
- **`ProductData`** (특산물): 이름, 아이콘, basePrice, **`isSpecial` 플래그**(미션 전용), 원산지 항구.
- **`ShipData`** (배): 이름, cannonPower(1~30), speed(1~10), cargoCapacity(10~1000), maxDurability(10~200), basePrice, 명성 게이트.
- **`DiscoveryData`** (발견물): 이름, 카테고리, 좌표, 일러스트, 해설, 관련 국가, **미션 전용 발견 가능**.
- **`MissionTemplate`** (의뢰): 발급 항구, 목표 발견물, 보상. **2026-05-31 단순화 — 발견물 의뢰만. 교역 의뢰는 기획에서 제거.** 특산물 매매는 시장(Market)에서 자유 매매로만.
- **`RegionData`**: 여러 항구/발견물을 묶는 지역. 잠금/해제 상태.

캐릭터·NPC SO:
- **`CharacterData`**: 능력치(용기·항해·눈썰미), 성별(외형), 초상화, 시작 명성. 플레이어/NPC 공통.
- **`NpcDefinition`**: CharacterData + 유형(Merchant/Escort/Pirate), 본거지 항구, 고용 보너스, 고용가.

런타임 상태 (SO 아님, 저장 데이터):
- `SaveSlot`: 플레이어 명성/돈/보유 배/현재 미션/도감/일반화된 특산물 목록/게임 시간/NPC 풀 상태.
- `MarketSnapshot`: 입항 시점에 결정되는 항구의 시세표(±20%).

### 5.2 콘텐츠 작성 가이드라인 (별도 문서로 분리 예정)
- 텍스트는 **초등 저학년(1~3학년) 어휘 수준** 기준. 한 문장 짧게, 어려운 한자어/외래어 회피.
- 한 발견물·항구의 메인 해설은 **2~4문장**. 더 알고 싶으면 "더 보기" 로 별도 페이지.
- 역사 서술은 **연도·인물 암기보다 "왜 그런 일이 있었는지" 위주.**
- 종교/식민지/전쟁 등 민감 주제는 사실 기반으로 중립적 서술.
- 자료 출처를 메타데이터에 기록 (저작권 안전 / 추후 검증 용이).
- 시대 고증: 15~17세기 기준. 그 시대에 존재하지 않은 사물/지명/국가명은 사용 금지.

### 5.3 다국어
- 1차는 한국어 전용.
- 단, 모든 텍스트는 처음부터 **키-값 분리**로 작성 (예: `port.lisbon.name`, `port.lisbon.description`). ScriptableObject에 직접 한글을 박아도 되지만, 키 기반 로컬라이즈 시스템(Unity Localization 패키지)을 처음부터 도입하면 추후 영어/일본어 확장이 무료에 가까워진다.

### 7.1 Asset Store 에서 에셋 받는 정확한 흐름

> Unity 6의 Asset Store는 **에디터 안에 카탈로그가 없다.** 브라우저에서 "내 계정에 추가" 한 뒤, 에디터의 Package Manager에서 다운로드/임포트하는 2단계 흐름이다.

1. **브라우저에서 획득**
   - https://assetstore.unity.com 접속 → Unity 계정 로그인(에디터에서 쓰는 것과 같은 계정).
   - 원하는 에셋 페이지에서 **`Add to My Assets`** (무료) 또는 **`Buy Now`** (유료) 클릭.
   - 이 단계가 끝나면 해당 에셋이 내 라이브러리에 등록됨.

2. **Unity Editor에서 다운로드 / 임포트**
   - `Window` → `Package Manager` 열기.
   - 좌상단 드롭다운을 **`Packages: My Assets`** 로 변경.
   - 방금 추가한 에셋이 목록에 보임 → 선택 → 우하단 **`Download`** → 끝나면 **`Import`**.
   - Import 창이 뜨면 필요한 항목만 체크해서 임포트 (예시 씬은 빼도 됨).

3. **임포트 위치 정리 (중요)**
   - 대부분의 Asset Store 패키지는 `Assets/` 루트에 자기 폴더를 만들어 떨어진다 (예: `Assets/JoystickPack/`).
   - 그대로 두면 프로젝트 루트가 어지러워지므로, 임포트 직후 **`Assets/ThirdParty/<에셋이름>/`** 으로 옮길 것을 권장.
   - 옮길 때 Unity 에디터에서 드래그하면 GUID·메타 파일이 같이 따라가서 안전. 파일 탐색기로 옮기면 참조가 깨질 수 있음.
   - 임포트 직후 git에는 `Assets/ThirdParty/<에셋이름>.meta` + 내부 파일들이 같이 커밋되어야 함 (그래야 다른 환경에서 동작).

4. **내 게임 코드에서 사용**
   - `Assets/Game/...` 의 스크립트/프리팹이 `Assets/ThirdParty/...` 의 자산을 참조하는 식으로 사용.
   - `Game.asmdef` 가 ThirdParty 어셈블리를 참조해야 하는 경우는 `asmdef`의 References에 추가.

5. **라이선스 확인**
   - 모든 Asset Store 에셋은 기본적으로 [Asset Store EULA](https://unity.com/legal/as-terms) 의 라이선스를 따른다. **상업 배포 가능**한 것이 대부분이지만, 일부는 "한 명의 시트(seat)당 1라이선스" 등 제약이 있음.
   - 어린이용으로 배포한다면 에셋 페이지의 **License** 섹션을 반드시 확인.
   - 받은 에셋의 이름·버전·라이선스·출처 URL을 `Assets/Game/Art/CREDITS.md` 에 한 줄씩 기록.

- **왼손 영역**: 가상 조이스틱 (조타). 왼쪽으로 밀면 좌현, 오른쪽으로 밀면 우현.
- **오른손 영역**: 위쪽 화살표 = 가속, 아래쪽 = 감속, 가운데 = 정지 버튼(돛 내림).
- **상단 HUD**: 메뉴 / 항해일지(도감) / 지도(미니맵 확장) 아이콘.
- 모든 터치 타깃은 **최소 48dp (≈ 9mm)** 이상. 어린이 손가락 + 두꺼운 손가락에도 잘 눌리도록.
- 양손 엄지 영역은 **Safe Area 안쪽**으로 둠. 노치/펀치홀과 겹치지 않도록.

### 8.2 입력 액션 (Input System 재작성)
기존 `Assets/InputSystem_Actions.inputactions` 의 `Move` / `Look` / `Attack` / `Interact` / `Crouch` 는 이 게임에 맞지 않으므로 새로 만든다. 모든 액션은 **터치 디바이스** 바인딩만 갖는다.

| 새 액션 | 타입 | 바인딩 |
|---|---|---|
| `Steer` | Value (Vector2) | `On-Screen Stick` (좌측 조이스틱 UI) |
| `Throttle` | Value (float, -1~1) | `On-Screen Button` ▲(+1) / ▼(-1) 두 개 합성 |
| `Confirm` | Button | 터치 (UI 버튼 또는 항구/발견물 패널의 탭) |
| `OpenJournal` | Button | HUD의 📖 버튼 |
| `OpenMap` | Button | HUD의 🗺 버튼 |
| `PauseStop` | Button | HUD의 ⏸ 버튼 (배 정지) |

`On-Screen Stick` / `On-Screen Button` 은 **Unity 공식 Input System 패키지에 포함**되어 있음 — 별도 결제 불필요. UI에 컴포넌트 붙이면 곧바로 액션과 연결됨.

### 8.3 조작 보조 (초등 저학년 배려)
- 약한 관성. 멈춤이 너무 길어지지 않도록 자연 감속 추가.
- "정지" 버튼을 한 번 누르면 배가 천천히 멈춤(돛 내림 연출).
- 미니맵에 가까운 항구/발견물 **방향 화살표** 표시.
- 육지에 부딪혀도 **데미지/사망 없이** 살짝 튕기고 끝. 시각·청각 피드백은 가볍게.
- 첫 진입 시 **튜토리얼 오버레이** — "여기를 밀어서 방향을 바꿔보세요" 식의 손가락 가이드.

### 8.4 기기별 고려
- **태블릿** (예: iPad 9.7" 이상): 조이스틱·버튼이 화면에 비해 작아지지 않도록 **DP/pt 기준 고정 크기** 사용 + 양쪽 끝 여백 자동 확대.
- **폰** (5.5" 이상): 가로 모드에서도 조이스틱/버튼이 화면 절반을 차지하지 않도록 적당히 작게.
- **반응형**: Canvas Scaler `Scale With Screen Size`, Match `Width or Height = 0.5` 정도가 무난한 시작점.

## 9. 진행도 / 저장

긴 모험으로 결정되었으므로 저장 시스템이 1차 마일스톤부터 견고해야 한다.

- **저장 대상**: 선택한 국적, 방문한 항구 ID 목록, 발견한 항목 ID 목록, 해제된 지역 목록, 마지막 위치 / 시간, (도입 시) 자금·도감 정렬 상태.
- **저장 방식**: `Application.persistentDataPath/save_slot_<n>.json` 형태로 1~3 슬롯. `PlayerPrefs` 는 작은 전역 설정(음량, 자막 크기 등)에만 사용.
- **저장 트리거 (자동 저장)**:
  - 항구 입항 직후
  - 발견물 획득 직후
  - 큰 메뉴(도감, 메뉴) 진입 직전
  - 앱 일시정지 (`OnApplicationPause(true)`) — 어린이가 갑자기 다른 앱으로 빠지는 경우 대비
- **버전·마이그레이션**: JSON 최상단에 `"version"` 필드를 두고, 게임 업데이트로 데이터 구조가 바뀌면 로드 시점에 마이그레이션. Play 출시 후엔 무시할 수 없음.
- **백업**: 11.4 참고. Android 자동 백업 기본값 유지(true) — 민감 정보 없으므로 OK.
- 어린이용이므로 **자동 저장 우선** (수동 "저장" 버튼 없음). 슬롯 선택은 메인 메뉴에서만.

### 11.1 AndroidManifest — 권한 / 네트워크 봉인

빌드 후 생성되는 manifest에 다음을 보장:

- `<uses-permission android:name="android.permission.INTERNET" />` **포함되지 않아야 한다.**
  - Unity는 일부 패키지(특히 Analytics, Ads, Cloud Build, 일부 네트워크 유틸)를 추가하면 자동으로 INTERNET 권한을 끌어들인다. 그런 패키지를 **애초에 깔지 않는 것**이 최선의 방어.
  - 보험으로 `Assets/Plugins/Android/AndroidManifest.xml` 에 명시적 제거 선언을 둘 수 있다:
    ```xml
    <manifest xmlns:android="http://schemas.android.com/apk/res/android"
              xmlns:tools="http://schemas.android.com/tools">
        <uses-permission android:name="android.permission.INTERNET" tools:node="remove"/>
        <uses-permission android:name="android.permission.ACCESS_NETWORK_STATE" tools:node="remove"/>
    </manifest>
    ```
- 카메라/마이크/위치/연락처/저장소 등 **어떤 위험(dangerous) 권한도 요청하지 않는다.**
- 외부 저장소 쓰지 않음 (Scoped Storage 이슈 자체 회피). `Application.persistentDataPath` 사용 — 앱 전용 영역, 권한 불필요.

> 각 마일스톤은 "다음 마일스톤이 데이터 추가만으로 가능한 상태"로 끝나는 것을 목표로 한다.

#### M1 — 항해 한 사이클 (포르투갈 고정)
- 리스본 ↔ 세우타 + 지브롤터 해협 발견.
- 시장(특산물 매매) 최소 흐름 + 모험가 조합의 발견물 의뢰 1건.
- 발견물 탐색(정박 및 탐색) → 의뢰 항구 복귀 → 돈 + 좋은 명성 보상.
- 데이터 구조는 메카닉 풀세트를 가정해 설계하되, 실제 데이터는 포르투갈/리스본/세우타만.
- **M1에서 제외**: 전투, NPC 풀(100명), 광장(고용), 조선소(매매/개조), 호위선/해적선. 메카닉 검증에 필수가 아니므로 M2 이후로.
- S23 실기에 `.apk` 설치해 손가락 조작감 확인.

#### M2 — 8개국 선택 + 각국 시작 항구
- 메인 메뉴에 국적 선택 화면 추가.
- 8개국 각 `NationData` 작성, 시작 항구 8개의 `PortData` 작성 (최소 한 줄 설명 + 임시 일러스트).
- 저장 슬롯에 선택 국적 기록.
- "내부 테스트(Internal Testing)" 빌드 첫 업로드 시점 후보.

#### M3 — 지역 확장 + 메카닉 풀세트
- 각 시작 항구를 중심으로 2~3개의 인접 항구 추가.
- 발견물 카테고리 4종(랜드마크/동식물/유적/사건)별로 최소 5개씩 도감 등록.
- `RegionData` 의 잠금 해제 시스템 작동.
- 도감 UI 완성.
- **메카닉 풀세트 도입**: 100명 NPC 풀, 광장(고용 최대 10명), 조선소(배 매매·수리), 시세 ±20% 변동, 전투 시스템(자동 전투 권장, §6.1 어린이 톤 적용), 명성 시스템 양방향.
- §8.14 "나쁜 명성 시스템 처리"는 M3 진입 전에 결정 필요.

**항해 길잡이 — 미니맵 / 큰 지도 / 줌 (M3 작업 항목):**

세계 가로가 5400 unit (≈ 40,000 km) 으로 매우 크다. 어린이가 길을 잃지 않게 길잡이 UI 가 사실상 필수. §1.1·§8.1·§8.3 에서 원칙적으로 언급됐던 것을 정식 작업 항목으로 등록.

- [ ] **미니맵 (우상단 RawImage)**:
    - 베이크된 세계지도 텍스처 (또는 별도 축소판) 를 RawImage 로 표시.
    - 플레이어 좌표를 lat/lng → UV 로 환산해 위치 마커 (배 아이콘) 를 실시간 갱신.
    - 가까운 항구·활성 의뢰 목적지에 작은 점·화살표 표시.
    - 1차 구현: 정적 텍스처 + 플레이어 마커만. 2차에서 항구 마커 추가.
    - 크기 권장: 화면 폭의 약 1/5 (S23 가로 기준 ~370px), 우상단 안전영역 안쪽.
- [ ] **큰 지도 모드 (`OpenMap` 액션)**:
    - 상단 HUD 의 🗺 버튼으로 토글 — 입력 액션은 이미 `OpenMap` 으로 정의됨 (§8.2).
    - 전체 화면을 덮는 대형 지도 UI. 발견한 항구·발견물만 표시 (도감 진행과 연동).
    - 핀치 줌 / 두 손가락 팬으로 확대·이동. 항구 아이콘 탭으로 정보 팝업.
    - 어린이 친화: "현재 위치" 버튼으로 즉시 플레이어 중심으로 복귀.
- [ ] **항해 카메라 줌 인/아웃**:
    - 핀치 줌 (멀티터치) 또는 +/- 버튼으로 메인 카메라 거리 조절.
    - 범위 제한 (너무 가깝게/멀리 X). 부드러운 보간으로 어린이 멀미 방지.
    - 줌 아웃 시 항구·발견물 마커 글자가 가독성 유지되도록 스케일 조정.
    - Cinemachine 사용 시 `CinemachineVirtualCamera.m_Lens.OrthographicSize` 또는 거리 파라미터 조정.

위 셋은 한 묶음으로 작업하면 좌표 → UV 변환·카메라 시스템을 공유할 수 있어 효율적.

**맵 시각·도구 보강 (M3 후반 또는 M3.5 후속):**
- [ ] **맵 색상 통일·일괄 변경**: 현재 `M3WorldMeshBaker.LandColor` 가 하드코딩된 단일 값. 머티리얼 한 곳에서 또는 별도 SO/팔레트로 분리하여 디자이너가 인스펙터에서 한 번에 변경 가능하게. 향후 대륙별 다른 색·고도별 그라데이션 확장 여지도 같은 구조에서.
- [ ] **바다 시각화**: 현재 `SeaPlane` 이 거의 단색 — 바다 느낌이 전혀 안 남. 옵션:
    - Natural Earth 의 Hypsometric + Water 텍스처 (가장 빠름)
    - URP Lit/Shader Graph 로 파도 노멀맵 + 살짝 움직임
    - 항해도 톤(고지도 갈색 종이 + 푸른 잉크) — 게임 콘셉트와 맞음
  - 어린이 게임 톤이므로 사실적 물보다 일러스트풍 추천.
- [ ] **에디터 모드: 항구 드래그 배치 도구**: 개발 중 항구 좌표를 시각으로 잡기 위한 에디터 전용 모드.
    - Scene View 에서 항구 마커를 드래그하면 즉시 위/경도 환산 → `PortData.latitude/longitude` 자동 갱신.
    - 메뉴 `Game ▸ Port Placement Mode` 토글로 켜고 끄기.
    - 한 번 잡은 좌표는 SO Asset 으로 영구 저장 → 런타임에는 평소처럼 사용.
    - 1:50m 해안선과 시각으로 매칭하면서 작업 가능 → 항구가 육지 안에 박히는 사고 방지.

### 12.3 짝 문서들 (Assets/Game/Data/)

본 문서(`GAME_PREP.md`)는 **시스템·정책·구조**를 다룬다. 분량과 검수 책임이 다른 두 문서가 별도로 분리되어 있다:

- → **[`Assets/Game/Data/GAME_MECHANICS.md`](Assets/Game/Data/GAME_MECHANICS.md)** — 게임 메카닉 / 시스템 규칙 (능력치·전투·NPC·도시·미션·시세). 사용자가 2026-05-29에 기획 초안 제공.
- → **[`Assets/Game/Data/CONTENT_DESIGN.md`](Assets/Game/Data/CONTENT_DESIGN.md)** — 실제 콘텐츠 데이터 워크북 (국가·항구·특산물·발견물·지역). 데이터 모델과 1:1 매핑되는 작성용 표.

세 문서의 관계: `GAME_PREP.md` 가 **무엇을·왜·어디서**, `GAME_MECHANICS.md` 가 **어떻게**, `CONTENT_DESIGN.md` 가 **무엇으로 채우나** 를 담는다.
